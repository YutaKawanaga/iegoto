## S-1 子どもプロフィールの昇格

**決定: 対応する。「メンバープロフィール」と「ログインアカウント」を最初からデータモデル上分離し、後からの紐づけ（昇格）・紐づけ解除（降格）を可能にする。**

### 仕様詳細
- エンティティを `Member`（家族内プロフィール）と `UserAccount`（Googleアカウントに対応するログイン主体）に分離する
- `Member.userAccountId` は nullable。null = ログイン不可のプロフィール（子ども等）
- **昇格フロー**: 大人が招待リンクを発行 → 子どもがリンクからGoogleログイン → 参加画面で「既存プロフィールに紐づける（自分は『長男』です）」または「新しいプロフィールを作る」を選択 → 既存を選ぶと過去の予定・担当履歴をそのまま引き継いでログイン可能になる
- **降格**: 家族設定画面から紐づけ解除が可能（プロフィールと予定は残り、ログインだけできなくなる）
- 制約: 1つの `UserAccount` が同一家族内で紐づけられる `Member` は1つまで（UNIQUE制約）

### 理由・トレードオフ
- 分離しておかないと後からの昇格はデータ移行を伴う大手術になる。分離のコストはJOIN1回分で軽微
- 「参加時に既存プロフィール選択」は招待リンクを踏んだ人なら誰でも任意のプロフィールに紐づけられるが、家族内の信頼を前提とするMVPでは許容（S-2の招待リンク保護で外部からの侵入を防ぐ）

## S-2 招待リンクの安全性

**決定: 「有効期限7日・失効可能・家族ごとに同時有効1本」の招待リンクとする。使用回数制限は設けない。**

### 仕様詳細
- トークン: 暗号論的乱数 256bit をURLセーフBase64で表現（推測不能）。DBにはハッシュ化して保存（漏洩対策、パスワードと同じ扱い）
- 有効期限: 発行から **7日間**（固定。設定UIは持たない）
- 使用回数: **無制限**（期限内なら家族全員分に使い回せる。数人規模の招待で回数管理はUX劣化に見合わない）
- 失効:
  - 手動失効（家族設定画面からいつでも）
  - 新しいリンクを発行すると旧リンクは自動失効（**同時に有効なリンクは家族ごとに最大1本**）
- 参加時の挙動: Googleログイン → 家族名を確認する参加確認画面 → 参加（S-1の既存プロフィール紐づけ or 新規作成）
- 参加が発生したら既存ログインメンバー全員にWeb Push通知（「◯◯さんが家族に参加しました」）— 意図しない参加の検知手段
- 不正対策: 無効・期限切れトークンへのアクセスは「リンクが無効です」のみ表示（家族名等の情報は一切出さない）

### 理由・トレードオフ
- 招待リンクは家族データ全開放の入口なので「短命・1本のみ・即失効可・参加通知」の4点で守る
- 回数制限やワンタイム化は安全性を上げるが、「夫婦+祖父母にまとめて送る」ユースケースを壊す。参加通知があれば侵入は即発覚し、メンバー削除で回復できる

## S-3 家族からの退出・メンバー削除

**決定: メンバーは論理削除（soft delete）とし、過去予定は残す。「退出（ログイン紐づけ解除）」と「プロフィール削除」を別操作として提供する。**

### 仕様詳細
- **退出**: `Member.userAccountId` を外すだけ。プロフィール・予定・履歴はすべて残る（例: 離婚・別居ではなく単にアプリをやめたい場合）
- **プロフィール削除**: `Member.deletedAt` を立てる論理削除
  - 過去の予定: 対象メンバー・担当者としての紐づけを保持したまま残す。表示は「（削除済み）名前」のグレー表示
  - 未来の予定・繰り返し予定の未来分: 対象メンバーからは自動で外す。**削除メンバーが唯一の担当者だった未来の予定は「担当者未定」に戻す**（F-04の担当者未定一覧に浮上し、家族が気づける）
  - 削除メンバーだけが対象だった未来の単発予定は削除せず残す（家族が個別に判断して消す。自動削除は事故のもと）
- 物理削除はMVPでは提供しない（家族アカウント自体の削除時にまとめて物理削除）
- 復元: 論理削除なので管理画面なしでも運用でリカバリ可能。UI上の復元機能はShould

### 理由・トレードオフ
- 「水泳教室にパパが送迎した」という過去の記録は家族の履歴であり、消すと予定一覧の意味が壊れる。匿名化は家族という少人数コンテキストでは意味がない（誰か推測できてしまう）ので採らない
- 唯一のリスクは削除メンバー名が残り続けることだが、家族内データなのでプライバシー問題にはならない

## S-4 タイムゾーン

**決定: MVPは Asia/Tokyo 固定。ただしデータモデルにはタイムゾーンカラムを最初から持たせ、将来の多TZ対応の芽を残す。**

### 仕様詳細
- 時間指定予定: DBにはUTCで保存（`timestamptz`）+ `Event.timezone`カラム（MVPでは常に `'Asia/Tokyo'` を保存）
- 終日予定: **日付のみ（`date`型）で保存し、TZ非依存とする**（終日予定をUTC変換すると日付ズレ事故が起きる典型パターンを回避）
- RRULE展開: `Event.timezone` のローカル時刻で展開してからUTCに変換する（DST・オフセット変化に正しく追従する唯一の方式。JSTはDSTがないが、実装をこの方式に統一しておく）
- フロントの表示・入力はすべて Asia/Tokyo 固定。ブラウザTZは参照しない（海外出張中でも家族の予定は日本時間で見えるのが正しい挙動）
- Googleカレンダーインポート: 元イベントのTZ情報を保持しつつ、表示はAsia/Tokyoに正規化

### 理由・トレードオフ
- ターゲット（日本の家族）にとって多TZ対応は価値ゼロでコストは大。固定が正解
- ただし「timezoneカラムを持たない」までやると将来対応が移行を伴うため、カラムだけは持つ（コストほぼゼロ）

## S-5 過去予定サジェストの仕様

**決定: 部分一致（前方一致優先）・直近1年・家族全員の予定を対象・最大5件・同一タイトルは最新1件に集約。**

### 仕様詳細
| 項目 | 仕様 |
|---|---|
| マッチ方式 | 部分一致。ソートは (1)前方一致 → (2)使用回数の多い順 → (3)最終使用日の新しい順 |
| 正規化 | 大文字小文字・全角半角・カタカナ/ひらがなを正規化して照合（「すいえい」で「水泳教室」は狙わない。読み仮名インデックスはCould） |
| 発火条件 | 1文字以上入力で発火、デバウンス300ms |
| 対象範囲 | **家族全員の予定**（家族は全データ共有が前提。ママが登録した「水泳教室」をパパも再利用できるべき） |
| 学習期間 | 直近1年（季節行事「プール開き」「発表会」を拾うため。それ以前は生活パターンが変わっている可能性が高い） |
| 候補数 | 最大5件（スマホの入力フォーム直下に出せる限界） |
| 集約 | 同一（正規化後）タイトルは最新使用の1件に集約して代表表示 |
| 引き継ぎ項目 | 開始・終了**時刻**（日付は入力中の日付を維持）、終日フラグ、対象メンバー、担当者、場所 |
| 引き継がない項目 | メモ（回ごとの個別性が高い）、繰り返し設定（意図しない繰り返し量産事故を防ぐ） |
| 除外 | Googleカレンダーからのインポート予定は候補に含めない（本アプリで編集できない予定を雛形にすると担当者等が空で引き継がれ混乱する） |

### 理由・トレードオフ
- 「繰り返し設定を引き継がない」が唯一非自明な点: サジェストの主用途は「不定期に繰り返す単発予定（通院、習い事の振替）」であり、真の定期予定はF-03の繰り返し機能でカバーされる。引き継ぐと1タップで無限繰り返し予定ができる事故導線になる

## S-6 通知の粒度設定

**決定: MVPはユーザーごとに2トグル（変更通知 / リマインダー）のみ。種類別・メンバー別のきめ細かい設定はShouldに送る。**

### 仕様詳細
- 設定はログインユーザー（`UserAccount`）単位。デフォルトは両方ON
  1. **予定の変更通知** ON/OFF: 他メンバーによる予定の作成・変更・削除の通知
  2. **リマインダー通知** ON/OFF: 自分が対象または担当の予定の開始前通知
- ノイズ抑制ルール（設定ではなくプロダクト側の仕様として固定）:
  - 自分が行った操作は自分に通知しない
  - 繰り返し予定の一括編集は1通知に集約（展開後の各回で通知しない)
  - Googleカレンダー同期による変更はプッシュ通知しない（1時間ごとに大量通知が来る事故を防ぐ。画面上の表示更新のみ）
  - 買い物リストのアイテム追加・チェックは通知しない（リアルタイム反映で画面には映る。通知はうるさすぎる）
- リマインダー: 予定ごとに「なし / 開始時 / 10分前 / 30分前 / 1時間前 / 1日前」から選択（デフォルト30分前、終日予定はデフォルト前日20:00）。通知先は対象メンバー+担当者に紐づくログインユーザー
- Should送り: 種類別設定（作成のみ/変更含む）、メンバー別ミュート、時間帯ミュート（おやすみモード）

### 理由・トレードオフ
- 通知設定は増やすほど「設定した記憶がなく通知が来ない」事故が増える。MVPは2トグル+賢いデフォルトに絞り、実運用（O-5の自家族2週間運用）でうるささを実測してから細分化する

## S-7 複数家族への所属

**決定: MVPは1ユーザー=1家族に制限する。ただしDBスキーマは複数所属可能な形（Member経由の間接参照）にしておき、制約はアプリケーション層+UNIQUE制約で表現する。**

### 仕様詳細
- `UserAccount` と `Family` の関係は `Member` テーブル経由（`Member` が familyId と userAccountId を持つ）
- MVPでは `Member.userAccountId` に **部分UNIQUE制約**（`WHERE user_account_id IS NOT NULL AND deleted_at IS NULL`）を張り、1アカウント1家族を強制
- すでに家族に所属しているユーザーが別の招待リンクを踏んだ場合: 「すでに◯◯家に参加しています。移籍するには現在の家族から退出してください」と案内（自動移籍はさせない）
- 将来複数所属を解禁する場合: UNIQUE制約の変更+「現在表示中の家族」切替UIの追加で対応可能（スキーマ変更不要）

### 理由・トレードオフ
- 複数所属（実家との共有等）は本物のニーズだが、MVPに入れると全画面に「どの家族のデータか」の文脈切替が必要になり、UI・クエリ・通知の全てが複雑化する
- スキーマだけ複数対応形にしておくコストはゼロ（Cozi型モデルの自然な正規化と一致する）ので、退路だけ確保する
