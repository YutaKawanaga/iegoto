# iegoto 実装順序（MVP到達までのフェーズ分け）

作成日: 2026-07-18
ステータス: 決定案（セルフレビューでの指摘「MVPスコープが個人開発として重い」への対応）
関連: `docs/requirements.md` §5 / `04-operations.md` O-5

要件のMust（F-01〜F-09）は変えない。ただし全部を作り切ってから初めて使うのではなく、
**フェーズごとに自分の家族へ投入して価値検証を前倒しする**。各フェーズ末が「動く状態」になるよう切る。

## フェーズ0: 雛形（すべての前提）

- モノレポ雛形（T-6構成）+ CI（lint / typecheck / test / build）+ ホスティングセットアップ
  （当面はVercel + Neon + Scheduler 3ジョブ + 日次バックアップ: `10-vercel-hosting.md`。Terraformは凍結）
- 認証（Googleログイン+セッション。`07-backend-design.md` §8）
- **⚠️ iOS Web Pushスパイクをここで実施**（半日〜1日）: 素のPWA+push通知のデモページをiPhone実機で
  「ホーム画面追加→許可→受信」まで確認する。F-08/F-09は全設計の中で最も不安定な外部前提であり、
  フェーズ4まで問題を寝かせない。想定外の制約が見つかったら通知計画（S-6/O-5）をこの時点で見直す

## フェーズ1: カレンダーの最小核 → 家族投入開始

- F-01 家族・メンバー管理（プロフィールCRUD・招待リンク）
- F-02 月表示 + メンバーカラー（週表示・フィルタは後回し可）
- F-03 予定CRUD（**単発のみ**。繰り返し・サジェストは含めない）+ 対象メンバー/担当者の分離
- F-05 買い物リスト + ポーリング反映（T-2）
- F-06 今日のまとめビュー
- **ここで自分の家族に投入**。紙/TimeTreeとの併用でよいので実データを流し始める

## フェーズ2: 差別化機能

- F-03 繰り返し予定（RRULE・例外編集3択・マスタ分割）— 最難所。domain層のテーブル駆動テストを先に書く（O-4）
- F-03 過去予定サジェスト（S-5）
- F-02 週表示・メンバーフィルタ・並列表示の仕上げ
- F-04 担当者一覧・担当者未定一覧
- **O-5の2週間計測はこのフェーズ完了後から開始**（繰り返し予定なしの家族カレンダーは実運用に耐えないため、フェーズ1時点では計測しない）

## フェーズ3: 連携

- F-07 Googleカレンダー読み取りインポート（OAuth認可・毎時同期・手動同期）

## フェーズ4: 通知・PWA仕上げ

- F-09 PWA（manifest・オンボーディング導線）
- F-08 Web Push（変更通知・リマインダー配信 `next_reminder_at` 方式）
- 通知はO-5の成功指標（全員がPWAインストール+許可）に直結するため、フェーズ0のスパイク結果を反映して仕上げる

## 運用ルール

- フェーズをまたぐ先回り実装をしない（例: フェーズ1で繰り返しのためのカラムだけ先に使い始める、はDBスキーマ定義済み（03）なので不要な複雑化）
- 各フェーズ末にタグを打つ（`phase-1` 等）。O-5の判定はフェーズ4完了後の2週間で行う

## 実装状況 (2026-07-23 時点)

- フェーズ0〜2: **完了**。本番 (https://iegoto-drab.vercel.app) で稼働中
- フェーズ3 (F-07 Google同期): **当面見送り** (実運用で必要性を再評価してから着手)
- フェーズ4 (F-08 Web Push / F-09 PWA): **完了** (通知の種類別ON/OFF・リマインダー配信ジョブ含む)
- 変更点: F-04「担当」は実装後のドッグフーディングで「対象メンバーで表現できる」と判明し
  **UIレベルで廃止** (DB/APIは温存し可逆。PR #14)。差別化仮説の棄却として記録する
