# iegoto（家ごと）要件定義書 v1.2

プロダクト名: **iegoto** — 家族の予定・買い物・分担など「家の共有事」をまるごと集めるWebアプリ
リポジトリ: iegoto（個人リポジトリ / モノレポ）
作成日: 2026-07-17（v1.2 同日更新）
ステータス: 要件確定。残論点は §10 に集約（Claude Codeで継続）

## 1. コンセプト

**「家族の予定・買い物・分担を1か所に集める、家族専用のWebカレンダー」**

TimeTreeのような汎用グループカレンダーではなく、家族（夫婦・カップル・子どものいる世帯）に特化する。
汎用性を捨てる代わりに、家族生活に固有の課題を解決する:

- 「誰の予定か」がひと目でわかる（人単位の色分け）
- 「誰が対応するか」を予定に紐づけられる（送迎などの担当者割当）
- スマホを持たない子どもも予定の主体として扱える（プロフィール制）
- 予定以外の家族の共有事項（買い物リスト）も同じ場所で管理

## 2. ターゲットユーザー

| 区分 | 内容 |
|---|---|
| プライマリ | 子育て中の夫婦（子どもの習い事・学校行事・送迎分担の管理） |
| セカンダリ | カップル・夫婦のみの世帯（お互いの予定共有） |
| 利用環境 | Webブラウザ（PC / スマホブラウザ）。レスポンシブ対応 |

## 3. スコープ

### 3.1 スコープに含むもの（MVP）
- 家族単位のカレンダー共有
- 人単位の色分け・フィルタ表示
- 予定の担当者割当
- 共有買い物リスト
- 今日のまとめビュー
- Googleカレンダーからの読み取り専用インポート
- 変更通知

### 3.2 スコープに含まないもの（Won't / 将来検討）
- 予定ごとのチャット（家族はLINE等で会話するため不要と判断）
- 共有アルバム
- 位置情報共有（Webアプリと相性が悪い）
- 公開カレンダー・広告・課金
- ネイティブモバイルアプリ（将来のフェーズで検討）

## 4. アカウントモデル（重要な設計決定）

**Cozi型: 家族 = 1アカウント + メンバーはプロフィール**

- 「家族」を1つのアカウント単位とし、その中に複数の**メンバープロフィール**を持つ
- 各メンバープロフィールは、ログイン可能（メールアドレス紐づけ）／閲覧専用の区別なく作成できる
- **メールアドレスやスマホを持たない子どもも、プロフィールとして予定の主体になれる**
- ログインできる大人は、家族内のすべてのデータにアクセスできる（家族内での権限分離はMVPでは行わない）
- 招待は招待リンク（URL）で行い、リンクからGoogleログインで家族に参加する
- **認証はGoogleログインのみ**（メール+パスワードは持たない）。Googleアカウントを持たない大人は参加できないことを許容するトレードオフ。子どもはログイン不要のプロフィールなので影響なし

## 5. 機能要件

### 5.1 Must（MVP必須）

#### F-01 家族アカウント・メンバー管理
- 家族アカウントの作成（サインアップ）
- メンバープロフィールの追加・編集・削除（名前、テーマカラー、アイコン）
- ログイン用Googleアカウントのプロフィールへの紐づけ（認証はGoogleログインのみ）
- 招待リンクの発行・失効

#### F-02 カレンダー表示
- 月表示（デフォルト）
- 週表示
- 人単位の色分け表示（メンバーカラーで予定を描画）
- メンバーによるフィルタ（全員表示 ⇔ 特定メンバーのみ表示の切替）
- 同一時間帯の予定重複時は**並列表示**（横に並べる）

#### F-03 予定管理（CRUD）
- 予定の作成・編集・削除
- 項目: タイトル、日時（終日/時間指定）、繰り返し、対象メンバー（複数可）、担当者、メモ、場所
- **対象メンバー（誰の予定か）と担当者（誰が対応するか）を別項目として持つ**
  - 例: 「水泳教室」対象=長男、担当=パパ（送迎）
- 繰り返し予定: **内部表現はiCal RRULE準拠、UIはGoogleカレンダー相当の柔軟性**
  - 毎日 / 毎週（曜日指定可） / 毎月（日付指定 or 第n曜日） / 毎年 / カスタム（間隔指定）
  - 終了条件: なし / 終了日 / 回数
  - 例外編集: 「この予定のみ」「これ以降すべて」「すべて」の3択
- **過去予定サジェスト**: タイトル入力中に過去の予定から候補を表示し、選択すると時間・対象メンバー・担当者・場所も引き継いで入力補完する（TimeTreeの同機能の強化版）

#### F-04 担当者割当
- 予定に担当者（家族メンバー）を割当できる
- 自分が担当の予定を一覧で確認できる
- 担当者未定の予定を「担当者未定」として一覧表示できる（家族内の押し付け合い可視化）

#### F-05 共有買い物リスト
- リストの作成（複数リスト可: 食料品、日用品など）
- アイテムの追加・チェック（購入済み）・削除
- 誰が追加したかの表示
- リアルタイム反映（他メンバーの追加・チェックが即時に見える）

#### F-06 今日のまとめビュー
- ホーム画面として、当日の予定（家族全員分）・自分が担当の予定・買い物リストの未購入アイテムをまとめて表示
- 翌日の予定のプレビュー

#### F-07 Googleカレンダー連携（読み取り専用）
- Google OAuthによる連携認可
- 指定したGoogleカレンダーの予定を本アプリに読み取り専用で表示
- インポート予定は特定メンバーに紐づけ可能（例: パパの仕事カレンダー）
- 定期的な自動同期（1時間ごと目安）

#### F-08 通知（Web Push先行）
- 予定の作成・変更・削除時に他メンバーへWeb Push通知（FCM想定）
- 予定のリマインダー通知（開始N分前）
- メール通知はShouldに後送り

#### F-09 PWA対応（Web Push成立の前提）
- Webアプリマニフェスト + Service Workerを実装し、ホーム画面追加（インストール）可能にする
- **iOS SafariはPWAとしてホーム画面追加した場合のみWeb Pushが届く**ため、F-08の前提要件
- 初回ログイン後にホーム画面追加を促すオンボーディング導線を用意

### 5.2 Should（MVP後の早期対応）
- 予定の検索
- メール通知（Web Pushの許可を得られなかったユーザー向けフォールバック）
- 日次アジェンダメール（朝に今日の予定をメール配信）
- ToDo/家事リスト（担当者・繰り返し設定つき）
- Googleカレンダーへの書き出し（双方向同期の第一歩）
- 予定へのファイル・画像添付

### 5.3 Could（将来検討）
- 献立計画・レシピ管理（買い物リストとの連動）
- 誕生日・記念日トラッカー
- 空き時間の可視化（家族全員が空いている日を探す）
- 学校行事・持ち物メモのテンプレート
- オフライン閲覧（PWAキャッシュ強化。ホーム画面追加自体はF-09でMust化済み）

## 6. 非機能要件（概要）

| 項目 | 要件 |
|---|---|
| 対応ブラウザ | Chrome / Safari / Edge 最新版、スマホブラウザ含む |
| レスポンシブ | スマホ幅（375px〜）を第一優先でデザイン |
| リアルタイム性 | 予定・買い物リストの変更は数秒以内に他メンバーへ反映 |
| セキュリティ | 家族データは家族アカウント外から一切アクセス不可。通信はHTTPS |
| プライバシー | 子どものデータを扱うため、外部への情報提供なし・広告なし |
| 可用性 | 個人〜小規模利用を想定。まずは単一リージョンで可 |

## 7. 決定事項ログ

| # | 論点 | 決定 | 補足 |
|---|---|---|---|
| 1 | 通知手段の第一弾 | Web Push先行 | iOS対応のためPWA化をMustに昇格（F-09）。メールはShouldでフォールバック |
| 2 | 繰り返し予定の仕様 | 内部RRULE準拠 / UIもGoogleカレンダー相当 | TimeTreeの繰り返しの弱さを反面教師に、曜日指定・第n曜日・終了条件まで対応 |
| 2-2 | 過去予定サジェスト | Mustに追加（F-03） | TimeTreeの良い点を強化して踏襲。時間・メンバー・担当者ごと補完 |
| 3 | 認証方式 | Googleログインのみ | Googleカレンダー連携と一貫。パスワード管理を持たない |
| 4 | 予定重複時の表示 | 並列表示 | 家族カレンダーは同時間帯の並存が常態のため |
| 5 | Googleカレンダー連携 | 読み取り専用インポートから開始 | 双方向同期はShould以降 |
| 6 | アカウントモデル | Cozi型（家族=1アカウント+プロフィール） | 子どもをログイン不要の主体にできる |

### 10.3 仕様の詰め残し
| # | 論点 | 内容 |
|---|---|---|
| S-1 | 子どもプロフィールの昇格 | 子が成長してスマホを持ったとき、既存プロフィールにGoogleアカウントを後から紐づけてログイン可能にできるか（データモデルに影響するため早期に決めたい。対応する方針を推奨） |
| S-2 | 招待リンクの安全性 | 有効期限・使用回数制限・失効の仕様。家族データ全開放の入口なので要件レベルで明記したい |
| S-3 | 家族からの退出・削除 | メンバー削除時に過去予定の扱いをどうするか（予定は残す/匿名化/削除） |
| S-4 | タイムゾーン | 日本固定でよいか。RRULE展開時のTZ処理方針（MVPはAsia/Tokyo固定を推奨） |
| S-5 | 過去予定サジェストの仕様 | 前方一致/部分一致、候補数、学習対象期間（直近1年など）、家族全員の予定を候補にするか自分関連のみか |
| S-6 | 通知の粒度設定 | 全変更を通知するとうるさい。メンバーごとに通知On/Off、種類別設定（作成のみ/変更含む）をMVPに入れるか |
| S-7 | 複数家族への所属 | 1人が複数の家族アカウントに属せるか（例: 夫婦それぞれの実家との共有）。MVPは1人1家族に制限を推奨 |
